# UI 重布局实施计划

> 来源：《五四三二一》Demo UI 布局说明文档
> 当前目标：把现有可玩原型调整为“左侧信息栏 + 中央主游戏区 + 右上协议面板 + 底部交互栏”的横屏像素地牢 HUD。
> 实施原则：先改显示层和信息组织，不在同一轮重写核心玩法；怪物动态战斗作为后续高风险阶段单独处理。

---

## 1. 当前工程基线

现有工程已经具备 UI 重布局所需的数据来源：

| UI 数据 | 当前来源 | 状态 |
| --- | --- | --- |
| 地图格子、雷数、房型 | `scripts/systems/Minefield.lua` | 已有 |
| 玩家位置、移动、撤离 | `scripts/systems/ExtractionRun.lua` | 已有 |
| HP、战斗力、雷伤、怪物结算 | `scripts/systems/Combat.lua` | 已有 |

| 54321 协议等级 | `scripts/systems/Protocol.lua` | 已有 |
| 小地图 | `scripts/ui/MiniMap.lua` | 已有 |

| 主房间画面 | `scripts/scenes/DungeonRoom.lua` | 已有 |
| 主 UI、结算面板、输入 | `scripts/main.lua` | 已有 |

重布局主要改造 `main.lua` 的 HUD 组织方式，并把重复绘制逻辑逐步拆到独立 UI 模块。

## 2. 推荐默认决策

以下决策作为 Demo 实施默认值，后续策划如有变更再调整：

| 接口 | 默认决策 |
| --- | --- |
| 屏幕比例 | 16:9 横屏优先 |
| 左侧信息栏宽度 | 屏幕宽度 24%，最小 260px，最大 360px |
| 默认小地图 | 先保留全 15x15，若数字不可读再切 9x9 局部扫描图 |
| 小地图标题 | 区域扫描图 |
| 未探索特殊房 | 默认不显示 |
| 四角撤离点 | Demo 中可见或半可见，保留现状优先 |
| 状态栏战斗力 | 保留显示，因为当前怪物系统仍依赖战斗力 |
| 金币显示 | HUD 显示本局金币；菜单和结算显示全局金币 |
| 零件定位 | 风险资源，成功撤离完整带出，失败按保底处理 |
| 提示方式 | 左下常驻目标 + 底部即时交互提示 |
| 雷房触发后 | 允许通行，不重复触发 |
| 事件房 | Demo 只保留旅商 |
| 协议推进条件 | 探索房间数 |
| 协议等级 1 | 不直接失败，只强化撤离压力 |
| 撤离距离 | 先用曼哈顿距离 |
| 撤离确认 | 保留“撤离”和“继续探索”两个按钮 |
| 怪物战斗 | 先做 UI 表现增强，完整动态危险区域后移 |

## 3. 阶段 1：HUD 布局骨架

目标：建立稳定的四区布局。

布局结构：

```text
┌──────────────────────────────────────────────────────────────┐
│ 左侧信息栏              中央主游戏区                  右上协议 │
│ 区域扫描图              房间背景/玩家/怪物/宝箱        54321    │
│ 状态信息                搜索点/事件/撤离装置                   │
│ 当前目标与提示                                                   │
│ 底部：快捷栏 / 交互提示 / 撤离点距离                            │
└──────────────────────────────────────────────────────────────┘
```

实施任务：

1. 在 `main.lua` 中新增统一布局函数，例如 `ComputeHUDLayout(w, h)`。
2. 将 HUD 绘制拆成明确分区：
   - `DrawLeftSidebar`
   - `DrawProtocolPanel`
   - `DrawBottomBar`
   - `DrawRoomView`
3. 中央主游戏区避开左侧信息栏和底部栏。
4. 右上协议面板覆盖在中央区右上，不参与主画面排版。
5. 底部栏高度固定在 72px 左右，后续根据实际字号微调。
6. 先使用统一像素风深色半透明面板，不在本阶段细抠美术。

验收标准：

- 1280x720 下所有 UI 无重叠。
- 左侧小地图、状态、提示都在左栏内。
- 中央房间关键对象不被左栏、底栏、协议面板遮挡。
- 移动、搜索、撤离、打开地图功能保持可用。

## 4. 阶段 2：左侧信息栏

目标：让玩家快速理解“我在哪、我还能不能继续、当前收益多少”。

### 4.1 区域扫描图

实施任务：

1. 标题显示为“区域扫描图”。
2. 默认显示全 15x15。
3. 当前玩家位置使用黄色描边或高亮框。
4. 已探索安全格显示雷数。
5. 已标记格显示旗帜。
6. 已触发雷房显示红色危险图标。
7. 已探索特殊房显示图标：
   - 怪物房：怪物/危险图标
   - 宝箱房：宝箱图标
   - 事件房：事件/旅商图标
   - 撤离房：青绿色撤离图标
8. 数字优先级高于图标，图标不能遮挡数字。
9. 扫描图下方增加短图例：
   - `数字 = 周围 8 格雷险`
   - `怪物/宝箱/事件不计入数字`

验收标准：

- 15x15 全图中当前位置清楚。
- 雷数可读。
- 房型图标不会干扰核心扫雷数字。

### 4.2 状态信息

实施任务：

1. 显示生命：`生命 当前/最大`。
2. 显示本局金币。
3. 显示本局零件。
4. 显示战斗力。
5. 可选显示当前区域雷数。
6. BUFF 暂只保留最多 3-5 个小图标位置，不做完整详情面板。

验收标准：

- HP、金币、零件、战斗力始终可见。
- 本局资源和全局金币不混淆。

### 4.3 当前目标与提示

实施任务：

1. 当前目标先固定为：`搜刮物资，前往撤离点`。
2. 提示区一次显示一条短文本。
3. 提示按优先级覆盖：
   - 高：踩雷、撤离、失败、协议降级
   - 中：怪物、宝箱、事件
   - 低：基础教程
4. 开局提示：`数字表示周围 8 个区域中的雷险数量。`
5. 进入关键房型显示对应提示。

验收标准：

- 评委开局 30 秒内能理解小地图数字含义。
- 提示不会刷屏，也不会遮挡主操作区。

## 5. 阶段 3：右上 54321 协议面板

目标：把协议从普通状态文字升级为主题核心 UI。

实施任务：

1. 扩展 `Protocol.GetStatus()` 的返回信息：
   - `level`
   - `title`
   - `description`
   - `color`
2. 阶段文案：

| 等级 | 名称 | 描述 |
| --- | --- | --- |
| 5 | 正常作业 | 区域稳定，允许回收。 |
| 4 | 轻度警戒 | 异常读数上升。 |
| 3 | 风险作业 | 继续深入将提高收益，也提高风险。 |
| 2 | 返程建议 | 信标仍在服务中，暂时。 |
| 1 | 最终建议 | 撤离是建议，不撤离是自主选择。 |

3. 协议面板显示：
   - `54321 协议`
   - 大号等级数字
   - 阶段名称
   - 一句短描述
4. 协议降级时增加表现：
   - 数字短暂放大
   - 面板闪烁
   - 屏幕边缘轻微警示
   - 提示区显示：`协议降至 X：阶段名`

暂不实施：

- 协议影响怪物、雷房或收益数值。
- 协议等级 1 直接失败。

验收标准：

- 协议变化明显。
- 面板不遮挡主房间关键交互。

---

## 6. 阶段 4：底部栏与即时交互提示

目标：让玩家清楚当前可以按什么。

实施任务：

1. 底部中间显示当前交互：
   - 普通可搜索：`[F] 搜索`
   - 宝箱：`[F] 搜索物资箱`
   - 事件：`[E] 与旅商交易`
   - 撤离：`[E] 启动撤离信标`
   - 怪物：`[F] 战斗 / 离开房间`
   - 放大地图：`[M] 区域扫描图`
2. 底部右侧显示最近撤离点：
   - `最近撤离点：东北 距离 18`
3. 距离计算先用曼哈顿距离：
   - `abs(player.x - exit.x) + abs(player.y - exit.y)`
4. 如果后续有完整道具系统，再启用快捷栏；当前只保留轻量入口：
   - `背包 [B]`

验收标准：

- 进入不同房型时提示自动切换。
- 无可交互对象时不显示误导性操作。
- 撤离方向和距离随玩家移动更新。

## 7. 阶段 5：放大地图界面

目标：统一放大地图与“区域扫描图”的表达。

涉及文件：`scripts/ui/MapOverlay.lua`

实施任务：

1. 标题改为“完整区域扫描图”。
2. 保留已有功能：
   - 标记
   - 已访问安全房传送
   - 当前玩家位置
3. 增加图例区：
   - 数字 = 周围 8 格雷险
   - 旗帜 = 玩家标记
   - 宝箱 / 怪物 / 事件 / 撤离
4. 保持打开大地图时的暂停状态。
5. 点击传送规则不扩展，仍只允许传送到已访问安全房。

验收标准：

- 完整 15x15 地图可读。
- 图例不遮挡地图。
- 标记和传送行为与当前版本一致。

## 8. 阶段 6：撤离确认与结算界面

目标：让搜打撤收益和损失更清楚。

### 8.1 撤离确认

实施任务：

1. 到达撤离房并交互后弹出确认面板。
2. 面板显示：
   - 当前金币
   - 零件数量与折算金币
   - 已探索区域数
   - 当前协议等级
3. 按钮：
   - `撤离`
   - `继续探索`

验收标准：

- 玩家能明确知道现在撤离可获得多少。
- 继续探索不会丢失当前局状态。

### 8.2 成功结算

实施任务：

1. 显示撤离成功标题。
2. 显示：
   - 带出金币
   - 零件折算
   - 探索区域
   - 最终协议等级
   - 当前全局金币
3. 击败怪物数如果没有可靠统计，暂不显示。

验收标准：

- 全局金币只入账一次。
- 直接金币和零件折算不混淆。

### 8.3 失败结算

实施任务：

1. 显示失败原因。
2. 显示：
   - 金币保留
   - 零件损失
   - 可抢救 1 个零件时显示选择
   - 天赋保险金
3. 失败规则保持当前工程逻辑：
   - 金币是安全资产，失败自动保留
   - 零件是风险资产，失败默认丢失
   - 可抢救 1 个零件换金币

验收标准：

- 玩家能明确知道哪些保留、哪些损失。
- 保底选择不会重复结算。

## 9. 阶段 7：怪物战斗 UI 与玩法改造

该阶段风险最大，不建议和 HUD 重布局同时做。

当前工程怪物系统是“进入怪物房后战斗力检定 + VS 演出 + 逃跑窗口”。新 UI 文档描述的是“可穿越动态危险区域”，两者差异明显。

### 9.1 低风险 Demo 版

先只增强表现，不重写规则：

1. 怪物头顶显示名称和战力。
2. 加怪物状态条样式。
3. 进入怪物房提示：`可按 F 战斗，也可离开房间`。
4. 怪物击败后房间显示为已清理。
5. 小地图怪物图标变暗或变为已清理状态。

### 9.2 完整动态危险版

后续单独实现：

1. 怪物新增状态：
   - `hp`
   - `attackCooldown`
   - `warningTimer`
   - `attackShape`
2. 攻击类型先选圆形冲击波，比十字攻击更适配当前自由移动房间。
3. 玩家攻击键建议使用 `Space`，避免和 `F` 搜索/战斗冲突。
4. 离开房间后怪物 HP 先重置，降低状态保存复杂度。
5. 击败后掉落金币/零件，并标记房间清理。

验收标准：

- 玩家理解怪物房不是必须清场。
- 攻击预警可见，受伤原因明确。
- 离开、战斗、击败三条路径都不会卡状态。

## 12. 完成定义

UI 重布局第一轮完成标准：

1. 玩家进入游戏后能立即看到区域扫描图、HP、本局收益、协议等级。
2. 小地图数字规则通过图例或提示表达清楚。
3. 当前房型对应的交互提示明确。
4. 撤离确认能清楚展示当前收益。
5. 成功和失败结算能清楚展示带出/损失。
6. 原有核心闭环不回退：
   - 移动
   - 搜索
   - 踩雷扣血
   - 怪物房
   - 宝箱房
   - 事件交易
   - 撤离
   - 失败抢救条款
